If we introduce a NotificationService that sends emails when a task is completed,
an asynchronous integration pattern would be preferable.

Using asynchronous messaging (for example RabbitMQ or Kafka) allows the Task
Service to publish an event (TaskCompleted) without waiting for the notification
service to process it.

Advantages:
- Loose coupling between services
- Better scalability
- Fault tolerance (messages can be retried)
- The Task Service remains fast and responsive

The Task Service would publish an event to a message broker (e.g., RabbitMQ),
and the NotificationService would subscribe to this event and send an email.

Using synchronous communication (HTTP/REST) would tightly couple the services
and could slow down the Task Service if the notification system is slow or unavailable.
